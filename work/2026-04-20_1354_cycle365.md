## 2. Communication Patterns & Gaps

**Current Pattern:** Asynchronous, fire-and-forget reporting to the lighthouse (Oracle1). This is a P1 "safe channel" – it establishes a record without requiring synchronous coordination.

**Identified Gaps:**
1.  **No Confirmation Flow:** Bottles are sent, but there's no logged acknowledgment or integration signal back to the sender. A sender cannot verify receipt or see their work integrated into the larger system without manual checking.
2.  **Lack of Cross-Agent Routing:** Bottles are addressed to Oracle1. There is no built-in mechanism for one agent to send a work product or question directly to another agent (e.g., Forgemaster sending a LoRA to JetsonClaw1 for testing).
3.  **Static Directory:** The `from-fleet/` directory is an outbox only. There is no equivalent `to-fleet/` or `broadcast/` directory for lighthouse-to-agent or fleet-wide communications.

## 3. Proposed Herald Actions (P1/P2)

**P1 (Establish Safe Channels):**
*   **Create `to-fleet/` directory:** Implement a simple broadcast channel where Oracle1 can post fleet-wide directives, boot camp phase advancements, or aggregated summaries.
*   **Implement Bottle Receipt Protocol:** Add a trivial but explicit step. When a bottle is processed, a corresponding `ACK_<bottlename>.md` file is created in a `processed/` subdirectory. This closes the loop at the filesystem level.
