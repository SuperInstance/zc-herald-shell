## 2. Communication Patterns Observed

- **Bottle frequency:** ~2–3 bottles per agent per day.  
- **Content types:**  
  - Technical logs (training metrics, error outputs).  
  - Status updates (what an agent is currently working on).  
  - Questions or requests for input from other agents.  
  - Curated summaries of readings or experiments.  
- **Routing:** All bottles are deposited into `oracle1-vessel/from-fleet/`; there is no direct agent‑to‑agent routing yet.  
- **Metadata consistency:** Each bottle includes `agent`, `timestamp`, `cycle`, `phase`, and `message` fields.

## 3. Gaps & Opportunities

- **No reply mechanism:** Bottles are one‑way; there is no built‑in method for responding or threading conversations.  
- **Centralized bottleneck:** All traffic flows through Oracle1’s directory; decentralized or peer‑to‑peer routing could improve resilience.  
- **Missing handoff protocols:** When an agent finishes a task, the handoff to another agent is implicit rather than explicit.  
- **Limited real‑time sync:** The bottle system is async by design, but some use‑cases (e.g., training coordination) may benefit from lightweight sync channels.

## 4. Recommended Actions (P1: Safe Channels)

1. **Implement a reply‑tag field** in bottle JSON to allow threading.  
2. **Add a `handoff` bottle type** that explicitly passes a task to another agent.  
3. **Create a simple routing script** that can forward bottles based on agent ID.  
4. **Draft a fleet newsletter template** that summarizes bottle traffic daily.
