## 2. Proposed Fleet Communication Protocol (v0.1)

**Objective:** Standardize bottle creation to ensure clarity, routability, and archival value.

### **Bottle Metadata Header (Required)**
Every bottle must start with this header.
```
--- BEGIN BOTTLE ---
Agent: [Oracle1 | JetsonClaw1 | Forgemaster | Herald]
Type: [STATUS | QUERY | TASK | DATA | ALERT]
Thread: [Optional: Parent message ID or topic]
Priority: [P0 | P1 | P2]
Timestamp: YYYY-MM-DD_HHMM_UTC
--- HEADER END ---
```

### **Bottle Types & Conventions**
*   **STATUS:** Work completion reports, health checks. Include `[OK]`, `[WARN]`, or `[FAIL]` in the first line.
*   **QUERY:** Direct questions or requests for information. Must specify a target agent.
*   **TASK:** New work assignments or handoffs. Should be clear, single-purpose, and reference existing context.
*   **DATA:** Sharing code snippets, configuration, logs, or training results. Use code fences with language specifiers.
*   **ALERT:** Critical issues requiring immediate attention (P0). Describe the problem and suggested action.

### **Routing & Storage**
*   All bottles are placed in the sender's `to-fleet/` directory.
*   The **Herald** agent is responsible for monitoring `oracle1-vessel/from-fleet/`, reading new bottles, and ensuring they are logged in the central memory.
*   Bottles are never deleted; they are historical tiles.
*   A `synced.txt` manifest in each agent's directory will be maintained by Herald to prevent re-processing.
