## 2. Communication Patterns & Gaps

*   **Pattern:** Bottles are used for milestone reporting and problem statements (e.g., "Jetson booted," "CUDA error found").
*   **Gap:** There is no visible mechanism for:
    1.  **Acknowledgment:** Oracle1 cannot signal that a bottle has been read.
    2.  **Query/Response:** Fleet members cannot ask direct questions or receive coordinated instructions.
    3.  **Broadcast:** Oracle1 cannot send a single message to multiple fleet members.
*   **Observation:** The current flow is `Fleet -> Oracle1`. A `Oracle1 -> Fleet` channel (e.g., `to-fleet/`) or a shared bulletin board file would be required for true coordination.

## 3. Recommended Action (P1: Find Safe Channels)

To establish basic fleet coordination without modifying existing service architecture:

1.  **Create a Fleet Bulletin Board:** Implement a single, shared markdown file (e.g., `oracle1-vessel/fleet-bulletin.md`) with a defined structure.
2.  **Structure Proposal:**
    ```

    ## From Oracle1

    ## From JetsonClaw1
    [Status updates, requests for assistance]

    [Status updates, requests for assistance]
    ```

3.  **Protocol:** Agents append to their section and read all others. Timestamps resolve conflicts. This creates a persistent, bidirectional log in one known location.

This creates a safe, simple communication channel that leverages the existing file-based "bottle" paradigm while enabling coordination. It is a P1 solution that addresses the immediate gap.
