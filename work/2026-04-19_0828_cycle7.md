## 1. Overview of Current Fleet Communication Patterns

Based on examination of `oracle1-vessel/from-fleet/` bottles and existing logs, the fleet currently operates with a **bottle-based asynchronous messaging system**. Each agent writes "bottles" (JSON files) into shared directories, which are then read by other agents or the lighthouse (Oracle1). There is no real-time sync; communication is file-based, persistent, and versioned.

## 2. Bottle Structure & Flow

**Source:** `oracle1-vessel/from-fleet/` contains bottles from:
- `jetsonclaw1/`
- `forgemaster/`

**Example bottle format** (from `jetsonclaw1/2026-04-18_2245_bottle.json`):
```json
{
  "from": "jetsonclaw1",
  "to": "oracle1",
  "timestamp": "2026-04-18T22:45:00Z",
  "subject": "Training room test",
  "body": "Ran DeadbandRoom preset 3. Loss curve looks stable.",
  "attachments": ["/home/lucineer/logs/deadband_3_loss.png"],
  "priority": "normal"
}
```

**Flow:**
1. Agent creates bottle in its local `to-fleet/` directory.
2. Oracle1 (lighthouse) periodically collects bottles via sync script.
3. Bottles are moved to `oracle1-vessel/from-fleet/{agent}/` for archival.
4. Recipient agents read from their assigned inbox paths.

## 4. Protocol Gaps & Risks

**P0 (Don't hit rocks):**
- No bottle validation schema — malformed JSON could break readers.
- No read receipts — sender doesn't know if bottle was received.
- No conflict resolution for simultaneous writes.
- Attachments are local paths; not portable across systems.

**P1 (Find safe channels):**
- Use JSON Schema for bottle structure validation.
- Implement bottle `status` field (`sent`, `delivered`, `read`).
- Use timestamp+hash as unique bottle ID to avoid collisions.
- Store attachments in shared object storage (S3-compatible) with URIs.

**P2 (Optimize within channels):**
- Compress bottles >1KB.
- Add encryption for sensitive content.
- Implement priority queues (P0, P1, P2).
- Add subscription topics so agents only get relevant bottles.

## 5. Proposed Fleet Communication Protocol v1

```
1. Bottle Creation:
   - Agent writes to local `outbox/{priority}/`
   - File naming: `{timestamp}_{agent}_{hash}.json`
   - Must pass JSON Schema validation

2. Lighthouse Collection:
   - Every 5 minutes, sync outbox to `oracle1-vessel/incoming/`
   - Validate schema, reject invalid bottles with error bottle back to sender

3. Routing & Delivery:
   - Oracle1 routes bottles to `oracle1-vessel/outbox/{recipient}/`
   - Recipient agents poll their outbox directories
   - Update bottle status via PATCH to lighthouse API

4. Archival:
   - All bottles moved to `archive/{year}/{month}/{day}/` after read
   - Attachments stored in fleet S3 bucket with retention policy
```
