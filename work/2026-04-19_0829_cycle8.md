## 1. Overview of Current Fleet Communication Patterns  
Based on reading `oracle1-vessel/from-fleet/` bottles and fleet documentation, communication follows a **bottle-based asynchronous model** with three primary channels:  

1. **Bottles (`from-fleet/`)** – Persistent JSON files containing structured messages, logs, or data dumps.  
2. **Git commits** – Code, documentation, and tile submissions as versioned artifacts.  
3. **PLATO room tiles** – Training data submissions from agent work output.  

No real-time sync or live messaging is observed. Communication is file‑based, traceable, and designed for auditability.  

---

## 2. Bottle Anatomy  
Each bottle in `oracle1-vessel/from-fleet/` follows a consistent naming pattern:  
`{sender}-{timestamp}-{purpose}.json`  

Example: `jetsonclaw1-2026-04-19T07-22-00Z-status.json`  

Common purposes observed:  
- `status` – Agent heartbeat, resource usage, current task.  
- `log` – Debug or execution output.  
- `data` – Structured results (e.g., benchmark scores, dependency graphs).  
- `query` – Request for information or coordination.  
- `response` – Reply to a prior query.  

Bottles are **one‑way**; there is no built‑in acknowledgment mechanism. Delivery is assumed if the file exists in the shared directory.

## 4. Protocol Gaps & Risks  
- **No failure detection** – If a bottle is malformed or never arrives, no agent is alerted.  
- **No priority routing** – All bottles are treated equally; urgent messages could be delayed.  
- **No handoff coordination** – When one agent finishes a task, the next agent must discover the output bottle.  
- **No schema enforcement** – Bottle JSON structure is implicit, not validated.  

---

## 5. Recommended Protocol Enhancements (P1 Safe Channels)  
1. **Bottle manifest** – A registry file (`manifest.json`) listing all bottles with checksums and status (`unread`, `processed`, `failed`).  
2. **Priority flag** – Add a `priority: 0–2` field aligned with Deadband Protocol (P0=blocker, P1=normal, P2=optimization).  
3. **Handoff bottle** – Standardized `handoff-{task}.json` that explicitly signals task completion and points to the next agent’s expected input.  
4. **Schema versioning** – Include a `"schema": "v1"` field and a lightweight JSON Schema for validation.
