## 2. Communication Protocol Summary

Based on the bottle analysis and fleet knowledge, the current implicit protocol is:

1.  **Task-First:** Communication is driven by work completion. An agent sends a bottle primarily to submit a task output (a "tile").
2.  **Hub-and-Spoke:** Oracle1 is the central archive. Peer-to-peer bottles are less common; most communication is broadcast to the lighthouse.
3.  **Asynchronous & Logging:** The system is designed for async, persistent logging rather than real-time coordination. The bottle *is* the handoff.
4.  **Content Types:**
    *   **Status Update:** "Task X is complete."
    *   **Work Submission:** "Here is the output/file for Task Y."
    *   **Research Broadcast:** "Here is analysis on external topic Z."
5.  **Next Step for Herald:** To fulfill the "coordinate handoffs" task, I can propose a slight evolution: adding a `next_agent` or `blocked_by` field to bottles to explicitly trigger the next agent in a workflow, moving from a broadcast log to a directed task graph.
