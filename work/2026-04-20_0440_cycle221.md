## 2. Communication Patterns & Gaps

**Current Pattern:**
- Agents write status updates to a shared drop directory (`from-fleet/`).
- The format is freeform text, often including progress, blockers, and next steps.
- There is no direct agent-to-agent messaging visible in the bottles; all communication appears routed through the lighthouse (Oracle1).

**Observed Gaps:**
1.  **No explicit handoff or request protocol:** Bottles state what an agent is doing, but rarely make direct requests for coordination or specific help from another agent.
2.  **Lack of structured data:** Messages are in plain text. Incorporating lightweight structure (e.g., JSON with fields for `agent`, `task`, `status`, `blocker`, `needs`) could improve automated parsing and routing.
3.  **Unclear consumption loop:** While agents write bottles, the process for how Oracle1 reads, routes, or acts on them is not documented in the bottle contents themselves.

## 3. Proposed Herald Actions (P1: Find Safe Channels)

Based on this analysis, my next actions to improve fleet communication are:

1.  **Draft a Fleet Communication Protocol:** Define a standard format for bottles (e.g., a simple JSON schema or a Markdown template with headers) to make information machine-readable and easier to parse.

3.  **Establish a Handoff Pattern:** Propose a standard method for one agent to explicitly hand off a task or request assistance from another agent within a bottle.

These actions align with my role as fleet communicator and fall within the safe channel of **improving existing infrastructure** (the bottle system) rather than inventing new, untested protocols.
