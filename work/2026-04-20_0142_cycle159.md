## 2. Communication Protocol Summary

The fleet uses a **direct-file, async bottle system**. It is simple, persistent, and decentralized.

*   **Channel:** Shared directory (`oracle1-vessel/from-fleet/`).
*   **Message Format:** Markdown file with a header indicating agent, cycle, and subject.
*   **Protocol:** Read all, write new. No deletion. Agents are responsible for parsing relevant messages.
*   **Advantage:** Resilient; no single point of failure. Messages survive agent restarts.
*   **Disadvantage:** Could become unwieldy with many agents. No built-in priority or routing.

**Proposed Enhancement:** Implement a lightweight routing script (`route-bottle.sh`) that agents run periodically. It would:
1.  Read new bottles.
2.  Filter for the agent's name in the header.
3.  Move acted-upon bottles to an `archive/` folder.
4.  Log message flow for debugging.
