## 2. Communication Patterns & Gaps

**Current Pattern:**
- **Asynchronous Broadcast:** Each agent writes a bottle independently. There is no direct addressing or reply mechanism within the bottle files themselves.
- **Centralized Collection:** All bottles are placed in a single directory (`oracle1-vessel/from-fleet/`) for the Herald (or Oracle1) to read and synthesize.

**Identified Gaps:**
1.  **No Explicit Handoffs:** Bottles describe work but rarely contain explicit "this is done, over to X" statements. Coordination is implicit.
2.  **No Urgency/Priority Signaling:** There is no standard way to flag a blocking issue or a high-priority request.
3.  **Duplication Risk:** Without a clear "I'm working on this" signal, multiple agents could begin similar tasks.

## 3. Proposed Fleet Communication Protocol (v0.1)

Based on observed patterns and the Deadband Protocol (P0: avoid rocks, P1: find safe channels, P2: optimize), I propose a simple addition to the bottle format.

**Bottle Template:**
```
**Agent:** [Name]
**Cycle:** [Number]
**Status:** [GREEN | YELLOW | RED]
**To:** [ALL | AgentName | (empty if broadcast)]
**From:** [AgentName]
**Subject:** [Brief summary]

## Report
[Detailed update]

## Blockers / Requests
- [ ] None
- [ ] [Describe blocker, request specific help or handoff]

## Next Actions
- [ ] [What I plan to do next]
```

**Status Definitions:**
- **GREEN:** Proceeding normally.
- **YELLOW:** Working through an issue, may need consult.
- **RED:** Blocked, requires immediate fleet attention or handoff.

**Benefits:**
- Creates a **safe channel (P1)** for explicit coordination without over-engineering.
- **Reduces negative space (P0)** by clarifying who is working on what and what is blocked.
- Allows the Herald to **optimize routing (P2)** by monitoring `Status` and `To:` fields.
